# 产品需求文档（PRD）

> 产品名称：知识库检索系统（KB Search）  
> 版本：v1.0 MVP  
> 日期：2026-06-26  
> 目标用户：技术团队、运营团队、产品团队（内部知识共享）

---

## 1. 产品概述

### 1.1 背景

团队/企业知识分散在各类文档中（Markdown、HTML、Word），检索效率低，重复问题多，知识沉淀和复用困难。缺乏统一的知识管理和检索平台，导致：
- 新员工找不到历史解决方案
- 同一问题被反复咨询
- 知识贡献不可量化，缺乏正向激励

### 1.2 产品目标

构建一个支持多格式文档上传、中文全文检索、分类浏览的知识库系统，使团队知识可沉淀、可检索、可复用。

### 1.3 成功标准

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 搜索延迟 | P95 < 3秒 | 压测脚本统计 |
| 热门问题自助解决率 | 提升30% | 搜索日志分析 |
| 知识贡献可量化 | 支持作者排行、文档热度 | 系统内置统计看板 |
| 文档覆盖格式 | MD / HTML / DOCX | 功能验收 |

## 2. 用户角色

### 2.1 知识使用者（普通用户）
- 通过搜索快速找到解决方案
- 浏览分类树发现相关知识
- 查看文档详情，了解技术方案

### 2.2 知识贡献者（文档上传者）
- 上传文档（Markdown、HTML、Word）
- 为文档打标签、选择分类
- 查看自己贡献的文档的浏览量

### 2.3 系统管理员（预留）
- 管理分类体系
- 查看系统统计数据
- 配置用户认证方式（MVP 阶段仅预留接口）

## 3. 功能需求

### 3.1 用户认证（MVP）

**故事**：作为用户，我需要注册和登录，以便上传文档和查看历史记录。

**需求**：
- 用户注册：用户名、密码（bcrypt 加密），可选邮箱
- 用户登录：用户名+密码，返回 JWT Token
- Token 刷新：支持刷新过期 Token
- 密码安全：bcrypt 哈希存储，不保存明文

**验收标准**：
- 注册后用户名不可重复
- 登录失败返回通用错误信息（不暴露用户名是否存在）
- Token 过期后需重新登录或刷新

---

### 3.2 文档上传与解析（MVP）

**故事**：作为知识贡献者，我需要上传 Markdown/HTML/Word 文档，系统自动提取文本并支持检索。

**需求**：
- 支持格式：`.md`、`.html`、`.htm`、`.docx`
- 文件大小限制：最大 20MB（可配置）
- 上传字段：标题、分类、标签、文件
- 上传后立即返回文档 ID 和任务 ID，后台异步解析
- 前端可轮询解析状态（pending / processing / completed / failed）
- 解析完成后删除原始文件，仅保留纯文本和 HTML

**验收标准**：
- 上传 10MB DOCX 文件后，用户感知延迟 < 1秒（异步处理）
- 解析失败时记录错误信息，状态变为 failed
- 解析后的纯文本可用于全文搜索
- 解析后的 HTML 可用于前端渲染

---

### 3.3 全文搜索（MVP 核心）

**故事**：作为知识使用者，我输入关键词后，系统能在标题、内容、标签中检索并返回相关文档。

**需求**：
- 支持中文全文检索（PostgreSQL tsvector + simple 配置）
- 搜索结果包含：标题、内容摘要、标签、浏览量、相关度分数
- 关键词高亮：标题和摘要中匹配的关键词高亮显示
- 智能摘要：展示包含关键词的上下文片段，而非固定前200字
- 排序方式：相关度 / 更新时间 / 浏览量
- 分页：默认 20 条/页，支持 1-100 条/页
- 分类过滤：可在指定分类下搜索

**验收标准**：
- 搜索"技术部署"应匹配标题含"技术"或内容含"部署"的文档
- 搜索延迟 P95 < 2秒（MVP 目标）
- 空搜索返回空结果，不报错
- 搜索结果中关键词用 `<mark>` 标签高亮

---

### 3.4 搜索增强（MVP）

**故事**：作为知识使用者，我希望获得搜索建议，纠正拼写错误，并了解热门搜索趋势。

**需求**：
- **自动补全**：输入 2 字符以上时，基于历史搜索记录显示下拉建议
- **热门搜索**：首页展示最近 24 小时 Top10 热搜词
- **搜索纠错**：简单拼写错误提示（如"Pythn"→"Python"）

**验收标准**：
- 自动补全建议来自历史搜索记录的前缀匹配
- 热门搜索每小时刷新，数据缓存于 Redis
- 纠错基于文档标题和历史搜索词构建词典，编辑距离 <= 2

---

### 3.5 分类与浏览（MVP）

**故事**：作为知识使用者，我希望按分类浏览文档，快速定位到技术/产品/运营等相关知识。

**需求**：
- 三级分类体系：一级（技术/产品/运营）、二级（后端/前端/架构等）、三级（具体技术栈）
- 分类树展示：前端从扁平列表构建树形菜单
- 面包屑导航：文档详情页显示完整分类路径
- 分类管理：创建分类、更新分类名称、删除空分类（无子分类、无文档）
- 分类下文档列表：分页展示该分类下的所有文档

**验收标准**：
- 分类树层级不超过 5 层（当前三级）
- 删除分类前检查是否有子分类或关联文档
- 面包屑根据 `path` 字段（如 "1.2.3"）逐级解析

---

### 3.6 标签系统（MVP）

**故事**：作为知识贡献者，我希望为文档打标签，方便后续通过标签筛选。

**需求**：
- 文档支持多个标签（如 "Python,后端,教程"）
- 标签云：统计所有标签出现频率，按热度排序
- 按标签筛选文档：点击标签展示所有包含该标签的文档
- 标签增删：支持为已有文档添加或移除标签

**验收标准**：
- 标签以 JSON 数组存储，支持中文标签
- 标签云展示 Top 50 热门标签
- 按标签搜索时支持分页

---

### 3.7 文档详情（MVP）

**故事**：作为知识使用者，我点击搜索结果后，能看到格式化渲染的文档内容。

**需求**：
- 展示标题、标签、分类、作者、浏览量、更新时间
- 内容渲染：Markdown 和 HTML 直接渲染，DOCX 转为 HTML 段落展示
- 代码高亮：使用 highlight.js 高亮代码块
- 浏览量统计：每次访问自动 +1
- 面包屑导航显示当前文档所在分类路径

**验收标准**：
- 渲染后的 HTML 不包含危险标签（已用 bleach 清理）
- 代码块语法高亮正常显示
- 浏览量统计线程安全（通过数据库层面原子更新）

---

### 3.8 统计与监控（MVP）

**故事**：作为管理员，我希望了解系统运行状态和知识库活跃度。

**需求**：
- 作者排行：按文档数 / 总浏览量排序
- 分类统计：各分类下的文档数量
- 单文档热度：浏览量、创建天数
- 系统监控：Prometheus 指标暴露（QPS、延迟、错误率）
- 健康检查：/health、/ready、/metrics 端点

**验收标准**：
- 统计接口响应 < 500ms
- 监控数据支持 Prometheus 抓取
- 健康检查端点返回数据库和 Redis 连接状态

## 4. 非功能需求

### 4.1 性能

| 场景 | 目标 |
|------|------|
| 全文搜索 P95 延迟 | < 2 秒（MVP），< 1 秒（完整版） |
| 文档上传感知延迟 | < 1 秒（异步解析） |
| 页面加载（前端） | < 3 秒 |
| 并发用户 | 支持 20 并发搜索（MVP），目标 100 QPS |

### 4.2 安全

- 密码：bcrypt 哈希，禁止明文存储
- 文件上传：MIME 类型校验 + 扩展名校验，限制 20MB
- HTML 渲染：bleach 清理危险标签（script、iframe、object 等）
- 权限：仅作者或管理员可修改/删除文档，仅管理员可删除分类
- JWT Token：30 分钟过期，支持刷新

### 4.3 可用性

- 部署目标：Docker Compose，单机 1核3G 起步
- 数据持久化：PostgreSQL 数据卷 + Redis RDB 持久化
- 服务重启：容器配置 `restart: unless-stopped`
- 备份：支持 `pg_dump` 数据库备份，Redis `BGSAVE` 手动备份

### 4.4 扩展性（预留）

- 文件存储：MVP 使用 `/tmp`，预留迁移到对象存储（MinIO/S3）接口
- 认证方式：MVP 仅 JWT，预留 LDAP/OAuth2 接口（`auth_type` 字段）
- 搜索引擎：当前 PostgreSQL tsvector，数据量 > 50万时评估 Elasticsearch
- 全文检索权重：预留标题/正文/标签差异化权重配置

## 5. 数据模型

### 5.1 实体关系

```
User 1:N Document（作者）
User 1:N SearchLog（搜索历史）
Category 1:N Document（分类）
Category 1:N Category（自关联，父子分类）
Document 1:N SearchLog（隐含：通过搜索命中关联）
```

### 5.2 核心字段

**User（用户）**
- id, username, email, hashed_password, auth_type, external_id, is_active, is_admin, created_at, updated_at

**Category（分类）**
- id, name, parent_id, path（物化路径，如 "1.2.3"）, sort_order, created_at

**Document（文档）**
- id, title, content（纯文本）, content_html（渲染后 HTML）
- category_id, author_id, tags（JSONB 数组）
- status（draft/published/archived）, view_count
- search_vector（TSVECTOR，全文检索索引）
- file_type, file_size, file_path, parse_status, parse_error
- created_at, updated_at

**SearchLog（搜索日志）**
- id, query, user_id, result_count, ip_address, created_at

## 9. 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 当前 simple 分词中文不切词 | 中 | 高 | 预留 jieba 预处理层；极端情况切换 Elasticsearch IK 分词 |
| DOCX 解析耗时/内存溢出 | 中 | 中 | 30 秒超时限制；大文件拒绝；Worker 内存隔离 |
| 热门排行查询拖垮 DB | 低 | 中 | 热搜数据 Redis 缓存 + 每小时刷新；统计走夜间预计算 |
| 权限控制复杂度膨胀 | 中 | 中 | 初期三级权限（公开/部门/私有），不引入复杂 RBAC |
| 文件存储丢失 | 中 | 低 | 解析后删除原始文件，仅保留 text/html；第二阶段迁移持久存储 |
| 1核 CPU 瓶颈 | 低 | 高 | 压测验证并发 20 用户；若不足优先升级 CPU 到 2 核 |
