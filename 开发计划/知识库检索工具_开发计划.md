## 一、项目概述

### 1.1 背景与目标

团队/企业知识分散在各类文档中，检索效率低，重复问题多，知识沉淀和复用困难。

**成功标准**：
- 用户能在3秒内找到目标文档（P95搜索延迟 < 3s）
- 热门问题自助解决率提升（减少重复咨询）
- 知识贡献可量化，形成正向激励

### 1.2 约束条件

| 维度 | 约束 |
|------|------|
| 数据量级 | 初期1万篇文档，单篇<10MB，年增长50% |
| 并发 | 峰值QPS ~100（内部工具），未来可能到500 |
| 团队 | 2-3名后端 + 1名前端，无专职运维 |
| 预算 | 倾向开源方案，云资源月成本<2000元 |
| 上线时间 | MVP 6-8周，完整版3个月 |
| 部署环境 | 单机Docker起步，未来可能上K8s |
| 服务器 | 1核3G内存 / 50GB磁盘 |

### 1.3 技术选型

| 层级 | 选型 | 理由 |
|------|------|------|
| 后端框架 | FastAPI + SQLModel | 异步支持、自动生成OpenAPI文档、团队学习成本低 |
| 数据库 | PostgreSQL 15+ | 原生全文检索（tsvector/tsquery）、pg_trgm模糊匹配、JSONB存元数据 |
| 分词 | zhparser | 中文分词插件，无需额外服务 |
| 缓存/队列 | Redis 7 | 热门排行、搜索建议、Celery任务队列 |
| 文档解析 | mistune + BeautifulSoup4 + bleach + python-docx | 覆盖MD/HTML/DOCX三种格式 |
| 任务队列 | Celery | 异步文档解析，独立Worker容器 |
| 文件存储 | /tmp（MVP） | 容器映射宿主机/tmp，解析后删除原始文件 |
| 前端 | Vue3 + Element Plus | 组件丰富，admin类界面开发快 |
| 部署 | Docker Compose | 单机四容器：web / worker / db / redis |

### 2.2 容器资源分配（1核3G）

| 容器 | 内存限制 | CPU | 说明 |
|------|----------|-----|------|
| web (FastAPI) | 512MB | 共享 | 1个uvicorn worker |
| worker (Celery) | 512MB | 共享 | 2并发解析任务 |
| db (PostgreSQL) | 1GB | 共享 | shared_buffers=256MB |
| redis | 256MB | 共享 | 开启RDB持久化 |
| 系统预留 | ~700MB | - | 缓冲余量 |

### 2.3 核心数据模型

```python
# 用户表（预留SSO兼容）
class User(SQLModel, table=True):
    id: int = Field(primary_key=True)
    username: str = Field(index=True)
    auth_type: str = Field(default="local")  # local / ldap / oauth2
    external_id: str | None = None  # SSO系统用户ID
    created_at: datetime

# 分类表（技术/产品/运营三级）
class Category(SQLModel, table=True):
    id: int = Field(primary_key=True)
    name: str
    parent_id: int | None = Field(default=None, foreign_key="category.id")
    path: str  # 物化路径 "1.2.3"
    sort_order: int = Field(default=0)

# 文档表
class Document(SQLModel, table=True):
    id: int = Field(primary_key=True)
    title: str = Field(index=True)
    content: str  # 纯文本，用于全文检索
    content_html: str  # 渲染后的HTML
    category_id: int = Field(foreign_key="category.id")
    author_id: int = Field(foreign_key="user.id")
    tags: list[str] = Field(default=[], sa_column=Column(JSONB))
    status: str = Field(default="published")  # draft/published/archived
    view_count: int = Field(default=0)
    search_vector: str = Field(sa_column=Column(TSVECTOR))
    file_type: str  # markdown / html / docx
    file_size: int  # 字节
    created_at: datetime
    updated_at: datetime

# 搜索日志表（用于热搜统计）
class SearchLog(SQLModel, table=True):
    id: int = Field(primary_key=True)
    query: str = Field(index=True)
    user_id: int | None = Field(foreign_key="user.id")
    result_count: int
    created_at: datetime
```

## 三、MVP阶段（6周）

### 3.1 MVP范围

**包含**：
- ✅ Markdown / HTML / DOCX 上传与解析
- ✅ 全文检索（标题/内容/标签）+ 高亮 + 摘要
- ✅ 分类树浏览 + 面包屑导航
- ✅ 热门搜索排行 + 自动补全建议
- ✅ 用户认证（JWT）

**不包含**（第二阶段）：
- ❌ 版本历史
- ❌ 附件管理
- ❌ 个人收藏/知识库
- ❌ 知识贡献统计
- ❌ SSO/LDAP接入（预留接口）

### 3.2 周计划

#### W1：基础设施

| 任务 | 详情 | 产出 |
|------|------|------|
| 项目脚手架 | FastAPI + SQLModel + Alembic 迁移 | 可运行的基础项目 |
| Docker Compose | 四容器编排：web/worker/db/redis | `docker-compose up` 一键启动 |
| 用户认证 | JWT注册/登录/刷新，密码bcrypt加密 | 认证API可用 |
| 文件上传 | 接收MD/HTML/DOCX，存/tmp，限制20MB，MIME校验 | 上传接口返回task_id |

**风险检查点**：验证1u3g下四容器共存稳定性，监控内存占用。

#### W2：文档解析

| 任务 | 详情 | 产出 |
|------|------|------|
| Celery Worker | 独立容器，2并发，Redis队列 | 任务分发正常 |
| Markdown解析 | mistune同步解析，提取纯文本+HTML | MD上传秒完成 |
| HTML解析 | BeautifulSoup4 + bleach清理危险标签 | HTML安全渲染 |
| DOCX解析 | python-docx异步提取段落文本 | DOCX后台处理 |
| 状态轮询 | 解析状态API + 前端进度显示 | 用户可见处理进度 |

**风险检查点**：DOCX大文件（>10MB）解析超时，加30秒超时限制。

#### W3：搜索核心

| 任务 | 详情 | 产出 |
|------|------|------|
| zhparser安装 | PostgreSQL中文分词插件 | 中文分词可用 |
| tsvector触发器 | 文档增删改自动更新search_vector | 全文检索索引 |
| 搜索API | 按标题/内容/标签搜索，支持排序（相关度/时间） | 搜索接口可用 |
| 结果高亮 | 关键词高亮 + 内容摘要（前200字） | 搜索结果可读 |

**风险检查点**：zhparser分词效果验收，搜索"技术部署"应匹配标题含"技术"的文档。

#### W4：浏览体验

| 任务 | 详情 | 产出 |
|------|------|------|
| 分类树CRUD | 技术/产品/运营三级，递归查询 | 分类管理后台 |
| 面包屑导航 | 根据path字段生成 | 详情页显示路径 |
| 文档详情页 | MD→HTML预览 / HTML→清理后渲染 / DOCX→段落结构 | 三格式正常展示 |
| 标签系统 | 文档标签增删，标签云 | 标签筛选可用 |

**风险检查点**：分类树层级过深（>5层）性能下降，当前三级无风险。

#### W5：搜索增强

| 任务 | 详情 | 产出 |
|------|------|------|
| 搜索日志 | 每次搜索记录query/result_count | 数据积累 |
| 热门搜索 | Redis Sorted Set，每小时刷新Top10 | 首页热搜榜 |
| 自动补全 | 前缀匹配（ZRANGEBYLEX），基于历史搜索 | 输入框下拉建议 |
| 搜索纠错 | 简单编辑距离纠错（可选） | 拼写错误提示 |

**风险检查点**：Redis内存占用，超256MB时老数据LRU淘汰，可接受。

#### W6：生产就绪

| 任务 | 详情 | 产出 |
|------|------|------|
| 生产配置 | Gunicorn+uvicorn workers调优，Celery并发数 | 性能最优 |
| 日志规范 | 结构化JSON日志，统一格式 | 可对接日志系统 |
| 监控指标 | Prometheus指标暴露（请求数/延迟/错误率） | 基础可观测 |
| 部署文档 | 安装步骤、环境变量、常见问题 | 可交接运维 |
| 压测 | 并发20用户搜索，P95<2s，内存<2.5G | 性能达标 |

**验收标准**：
- 搜索P95延迟 < 2s
- 内存占用 < 2.5G（留500MB缓冲）
- 上传DOCX后用户感知延迟 < 1s（异步处理）

## 六、风险与应对

| 风险 | 概率 | 影响 | 预案 |
|------|------|------|------|
| zhparser中文分词效果差 | 中 | 高 | 预留jieba+自定义词典的Python预处理层；极端情况切到ES IK分词 |
| DOCX解析耗时/内存溢出 | 中 | 中 | 30秒超时限制，大文件拒绝；Worker内存隔离，不影响web服务 |
| 热门排行查询拖垮DB | 低 | 中 | 热搜数据Redis缓存+每小时刷新，非实时；统计走夜间预计算 |
| 权限控制复杂度膨胀 | 中 | 中 | 初期三级权限（公开/部门/私有），PostgreSQL RLS，不引入复杂RBAC |
| /tmp数据丢失 | 中 | 低 | 解析后删除原始文件，只保留text/html；第二阶段迁移持久存储 |
| 1核CPU瓶颈 | 低 | 高 | 压测验证并发20用户；若不足，优先升级CPU到2核 |

## 七、Docker Compose 配置

```yaml
version: '3.8'

services:
  web:
    build: .
    command: uvicorn main:app --host 0.0.0.0 --port 8000 --workers 1
    volumes:
      - /tmp/kb-uploads:/tmp/uploads
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgresql://kb:kb@db:5432/kb
      - REDIS_URL=redis://redis:6379/0
      - CELERY_BROKER_URL=redis://redis:6379/1
      - SECRET_KEY=${SECRET_KEY:-changeme}
    depends_on:
      - db
      - redis
    deploy:
      resources:
        limits:
          memory: 512M
    restart: unless-stopped

  worker:
    build: .
    command: celery -A tasks worker --loglevel=info --concurrency=2
    volumes:
      - /tmp/kb-uploads:/tmp/uploads
    environment:
      - DATABASE_URL=postgresql://kb:kb@db:5432/kb
      - REDIS_URL=redis://redis:6379/0
      - CELERY_BROKER_URL=redis://redis:6379/1
    depends_on:
      - db
      - redis
    deploy:
      resources:
        limits:
          memory: 512M
    restart: unless-stopped

  db:
    image: postgres:15-alpine
    environment:
      - POSTGRES_USER=kb
      - POSTGRES_PASSWORD=kb
      - POSTGRES_DB=kb
    command: >
      postgres
      -c shared_buffers=256MB
      -c max_connections=100
      -c effective_cache_size=512MB
      -c work_mem=4MB
    volumes:
      - pg_data:/var/lib/postgresql/data
    deploy:
      resources:
        limits:
          memory: 1G
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    command: >
      redis-server
      --appendonly yes
      --save "60 1"
      --maxmemory 256mb
      --maxmemory-policy allkeys-lru
    volumes:
      - redis_data:/data
    deploy:
      resources:
        limits:
          memory: 256M
    restart: unless-stopped

volumes:
  pg_data:
  redis_data:
```
